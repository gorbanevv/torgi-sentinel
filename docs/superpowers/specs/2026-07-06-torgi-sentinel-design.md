# Torgi Sentinel — дизайн системы

**Дата:** 2026-07-06
**Статус:** утверждён пользователем (дизайн), ожидает финального ревью спеки

## 1. Цель

Мониторить портал torgi.gov.ru (ГИС Торги, имущество Росимущества на торгах) и
**максимально быстро** (задержка 1–3 сек) присылать в личный Telegram уведомления
о появлении новых лотов по двум фиксированным фильтрам:

- **Фильтр A:** Севастополь + категория «Недвижимость».
- **Фильтр B:** Ростовская область + категория «Транспорт → Автомобили и мототехника».

Быстрее, чем любой человек или бот на этом источнике.

## 2. Ключевой технический факт (проверено вживую 2026-07-06)

Портал `torgi.gov.ru/new/public/lots/reg` — это SPA поверх **публичного JSON-API без
авторизации**:

```
GET https://torgi.gov.ru/new/api/public/lotcards/search
```

Проверенные параметры:

| Параметр | Значение | Назначение |
|---|---|---|
| `lotStatus` | `PUBLISHED` | только опубликованные лоты |
| `dynSubjRF` | код региона, напр. `61` | **фильтр по региону** (подтверждён: Ростов=61 → 25 лотов) |
| `catCode` | код категории | **фильтр по категории** (подтверждён как рабочий параметр) |
| `sort` | `firstVersionPublicationDate,desc` | самые свежие сверху |
| `size` | напр. `20` | размер страницы |
| `withFacets` | `true`/`false` | вернуть агрегаты категорий (`categoryFacet`) |
| `page` | 0-based номер страницы | пагинация (подтверждён вживую; `number` игнорируется) |

Формат ответа — Spring Data Page: `content[]`, `totalElements`, `totalPages`, `number`, `size`.
Каждый элемент `content[]` содержит: `id`, `noticeNumber`, `lotNumber`, `lotStatus`,
`biddType`, `biddForm`, `lotName`, `lotDescription`, `priceMin`, `biddEndTime`,
`lotImages[]`, `characteristics[]`, `attributes[]` (в т.ч. площадь и обременения),
`subjectRFCode`, `category{code,name}`, `createDate`, `timeZoneName`.

### Замер скорости (ключ к «быстрее всех»)
- Холодный запрос (новый TLS-хендшейк): **~3.3 сек**.
- Тёплый запрос (keep-alive соединение): **~0.23 сек**.
- Вывод: **держать соединение живым — обязательное требование**, это 14× ускорение
  и наше главное преимущество перед конкурентами, открывающими соединение заново.

### Почему опрос, а не push
У torgi.gov.ru **нет** вебхуков, публичного WebSocket/SSE или очереди событий. Открытые
данные обновляются раз в сутки. Единственный реалтайм-механизм — опрос API; так работают
все боты на этом источнике. «Быстрее всех» достигается за счёт: (1) низкого пинга (VPS в РФ),
(2) тёплого соединения, (3) частого опроса 2–3с, (4) мгновенной отправки в Telegram.
Скрытый push-эндпоинт будет проверен на этапе внедрения, но архитектура на него не
опирается.

## 3. Среда исполнения

- **Хостинг:** личный VPS пользователя на Timeweb (РФ). ВАЖНО (обновление):
  torgi.gov.ru блокирует IP дата-центров (Timeweb ✗, Cloudflare WARP ✗ — проверено).
  Рабочий бесплатный путь — запуск на резидентском/мобильном IP (Android+Termux).
  См. TERMUX.md. VPS остаётся запасным вариантом.
- **ОС:** Linux (уточнить дистрибутив при деплое; ориентир — systemd).
- **Доставка:** новый Telegram-бот (создаётся через @BotFather), уведомления в личку
  пользователю (один `chat_id`).
- **Локальная машина пользователя:** Windows, есть Node v25, Python — только заглушка
  Microsoft Store (настоящего нет). Разработка/деплой ориентируются на VPS, локально
  Python ставить не требуется.

## 4. Стек (обновлено: Node.js zero-dep)

- **Node.js (v18+, на VPS ставим LTS)** со **встроенным модулем `https`** — и опрос API,
  и вызовы Telegram. **Ноль внешних зависимостей** = нечему ломаться на VPS, нечего обновлять.
- **keep-alive `https.Agent`** — тёплое соединение (0.23с вместо 3.3с). Это ядро скорости.
- **Дедуп** — атомарный JSON-файл (write-temp + rename). Объём фильтров крошечный,
  SQLite избыточен; JSON-стор проще и переживает перезапуски.
- **Telegram Bot API** напрямую (`sendPhoto` / `sendMessage`) через `https`.
- **`node:test`** (встроенный раннер) — тесты без зависимостей.
- **systemd** — автозапуск/автоперезапуск, логи через journald.

Обоснование смены Python → Node (первоначально в спеке был Python): (1) на машине
пользователя нет настоящего Python (только заглушка Microsoft Store), а Node v25 есть —
значит код тестируется локально сразу; (2) весь ресёрч API уже проведён на Node
keep-alive против этого же эндпоинта — подход проверен вживую; (3) zero-dep максимизирует
надёжность долгоживущего демона. На скорость и стабильность влияет положительно.

## 5. Архитектура

Компоненты, каждый с одной ответственностью и явным интерфейсом:

```
Poller ×2 (по фильтру) ──▶ Deduper (SQLite) ──▶ Formatter ──▶ Telegram Notifier
   ▲ warm httpx client                                              │
   └── torgi.gov.ru API                                    личка пользователя
```

- **Poller** — по одному async-циклу на фильтр (2 шт.). Держит тёплый httpx-клиент,
  раз в 2–3с запрашивает `/lotcards/search` со своими `dynSubjRF`+`catCode`,
  `sort=firstVersionPublicationDate,desc`, `size=20`. Отдаёт список лотов в Deduper.
  Вход: конфиг фильтра. Выход: список карточек лотов. Зависимость: httpx-клиент.
- **Deduper (store)** — JSON-файл `data/seen.json` вида
  `{filters: {имя: {seededAt, seen: {lotId: firstSeenISO}}}}`; запись атомарная
  (tmp-файл + rename). Ключ лота — `id` карточки (`noticeNumber_lotNumber`).
  Возвращает подмножество «новых» лотов. При первом запуске засевается текущими
  лотами (см. §6). Вход: список лотов. Выход: только новые. Зависимость: файл JSON.
- **Formatter** — превращает карточку лота в сообщение (текст + опц. фото).
  Вход: карточка лота + тип фильтра. Выход: структура сообщения. Зависимостей нет
  (чистая функция).
- **Telegram Notifier** — отправляет сообщение с ретраями. Вход: структура сообщения +
  `chat_id`. Выход: факт доставки. Зависимость: Telegram Bot API, токен.

## 6. Поток данных и обработка новизны

1. Poller получает страницу свежих лотов (новейшие сверху).
2. Deduper отбирает пары `(noticeNumber, lotNumber)`, которых нет в SQLite.
3. Для каждого нового лота Formatter собирает сообщение (площадь, цена, даты, ссылка, фото).
4. Notifier отправляет в Telegram; при успехе Deduper помечает лот как виденный.
5. **Первый запуск (seed):** все текущие лоты по фильтру записываются в `seen_lots`
   БЕЗ отправки — чтобы не завалить пользователя существующим списком. Уведомления идут
   только по тому, что появилось *после* старта.
6. **Защита от пропущенной пачки:** если на первой странице все лоты новые (значит между
   опросами вышло > `size`), Poller листает следующие страницы, пока не встретит уже
   известный лот.

Ссылка на лот собирается как `https://torgi.gov.ru/new/public/lots/lot/<noticeNumber>_<lotNumber>`
(точный формат подтвердить при внедрении).

## 7. Надёжность

- **Тёплое соединение:** один httpx `AsyncClient` на фильтр живёт всё время; при обрыве —
  переподключение с сохранением цикла.
- **Адаптивный backoff:** на `429` / `5xx` / таймаут — экспоненциальная пауза
  (напр. 2→4→8→…→макс 60с), затем возврат к штатным 2–3с. Исключает бан по IP и падения.
- **Битый TLS:** сертификат портала не проходит стандартную валидацию — используем
  `verify=False` (публичный read-only гос-API, приемлемо).
- **Персистентный дедуп:** JSON-store на диске (атомарная запись) — после ребута VPS
  ничего не задваивается и не теряется; лот помечается «виденным» только после
  успешной отправки в Telegram.
- **Heartbeat:** запись в лог раз в N минут («жив, последний опрос N сек назад»).
- **systemd:** `Restart=always`, автозапуск при загрузке VPS.

## 8. Содержимое уведомлений

**Недвижимость (Севастополь):** название лота; **площадь** (из `characteristics`/`attributes`,
приоритетное поле); адрес/регион; цена (`priceMin`); вид торгов (`biddType`/`biddForm`);
дата окончания подачи (`biddEndTime`); ссылка на лот; **фото** из `lotImages[]` (если есть).

**Транспорт (Ростов):** название (марка/модель); регион; цена; вид торгов; дата окончания;
ссылка; фото.

Фото прикрепляется через `sendPhoto` (URL картинки строится из `lotImages[]` — точный
формат file-эндпоинта подтвердить при внедрении); при отсутствии фото — `sendMessage`.

## 9. Конфигурация

Единый конфиг (`config.json` по образцу `config.example.json`, поверх — переменные
окружения `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID`), не в гите:
- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`
- Список фильтров: `{name, dynSubjRF, catCodes[], real_estate: bool}`
- `POLL_INTERVAL_SECONDS` (по умолч. 3), `PAGE_SIZE` (20), лимиты backoff
- Путь к SQLite

## 10. Разрешённые параметры (все проверены вживую 2026-07-06/07)

| Что | Значение |
|---|---|
| Севастополь | `dynSubjRF=80` (подтверждён: лоты subjectRFCode=92, кадастр 91:…) |
| Ростовская область | `dynSubjRF=63` (подтверждён: subjectRFCode=61, кадастр 61:…) |
| Недвижимость | `catCode=7` — родительский узел (нежилые 11, жилые 9, здания 8, гаражи 10, незавершёнка 206, иное 208…) |
| Автомобили и мототехника | `catCode=100` — родительский узел (легковые 100001, грузовые 100002, мото 100000, автобусы 31); спецтехнику 101 и водный 16 НЕ включает |
| Мульти-`catCode` | **не поддерживается API** (возвращает 0) — на фильтр ровно один родительский код |
| Пагинация | параметр `page`, 0-based (`number` игнорируется) |
| Сортировка | `sort=firstVersionPublicationDate,desc` — подтверждена строгим убыванием дат |
| Страница лота | `https://torgi.gov.ru/new/public/lots/lot/{noticeNumber}_{lotNumber}` |
| Картинка | `https://torgi.gov.ru/new/file-store/v1/{lotImages[i]}` (200, image/jpeg, магия ffd8 проверена) |
| Карточка лота API | `https://torgi.gov.ru/new/api/public/lotcards/{id}` |
| Площадь | `characteristics[]`: код `Square*` / имя `*Площадь*` (напр. `SquareZU`=1281.0) |

Осталось уточнить при деплое: дистрибутив Linux на VPS и версия Node (нужна ≥18).

## 11. Область (что НЕ делаем — YAGNI)

- Нет веб-интерфейса, дашборда, БД-сервера, очередей.
- Нет произвольных пользовательских фильтров через бота — только два зашитых.
- Нет мультипользовательской рассылки — один получатель.
- Push-механизм не реализуем, пока не найден реальный эндпоинт.

## 12. Критерии успеха

- Новый лот по любому из двух фильтров доходит в Telegram за ≤ 3 сек после появления в API.
- Ноль дубликатов и ноль пропусков при штатной работе и после перезапусков VPS.
- Сервис живёт неделями без вмешательства (systemd auto-restart, backoff вместо падений).
- В недвижимости в сообщении всегда присутствует площадь (когда она есть в данных).
